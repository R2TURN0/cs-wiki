까마귀발(IE 표기법)의 '심플 버전'입니다
이 툴(ERD-Cloud 등에서 주로 사용하는 방식)은 화면을 깔끔하게 만들기 위해 전통적인 까마귀발 기호를 아주 단순화해서 표현한 것입니다.
원래 정석 까마귀발은 기호 끝에 동그라미(O), 세로줄(|), 삼지창(<) 등이 복잡하게 조합되지만, 이 툴은 핵심만 남겼습니다.
삼지창 모양 (<)이 붙은 쪽: 다(Many, N/M) 쪽을 의미합니다. (사진에서 OrderItem 테이블 쪽)
아무것도 없는 민자 직선 쪽: 일(One, 1) 쪽을 의미합니다. (사진에서 Order 테이블 쪽)
점선(Dashed Line): 비식별 관계를 뜻합니다. 부모의 PK가 자식의 일반 외래키(FK)로 들어간 상태입니다.
즉, 사진에 그려진 선은 "Order(1)와 OrderItem(N)은 1:N 비식별 관계다"라는 것을 이 툴만의 방식으로 표현한 것입니다. 까마귀발이 맞긴 맞는데, 발가락 다이어트를 조금 심하게 한 녀석이라고 보시면 됩니다.
2. 왜 양방향 화살표로 가지 않을까?
네트워크 망구성도나 프로그램의 데이터 흐름도(DFD)에서는 데이터가 주고받는 것을 표현하기 위해 양방향 화살표를 자주 씁니다. 하지만 DB ERD는 '데이터의 이동 방향'을 그리는 게 아니라, 테이블 간의 '구조적 종속 관계'를 그리는 도구이기 때문에 양방향으로 표현하지 않습니다.
물리적인 연결은 단방향입니다: 관계형 데이터베이스(RDB)에서 두 테이블이 연결되는 물리적인 이유는 딱 하나, 한쪽 테이블이 다른 쪽 테이블의 기본키(PK)를 외래키(FK)로 훔쳐 와서 저장(참조)하고 있기 때문입니다.
참조하는 주체는 자식입니다: 위 사진을 보면 OrderItem 테이블 맨 밑에 id (FK)가 있죠? 이는 OrderItem이 상위 개념인 Order를 일방적으로 바라보고(참조하고) 있다는 뜻입니다. Order 테이블 안에는 OrderItem에 대한 정보가 물리적으로 들어있지 않습니다.


식별관계는 아니면서 부모 데이터가 삭제되면 자식 데이터도 삭제되게 하는법

`ON DELETE CASCADE`

DA# 툴에서는 어떻게 설정하나요?
DA#이나 ERwin 같은 전문 모델링 툴에서는 이 설정을 마우스 몇 번 클릭으로 적용할 수 있습니다.
두 엔티티 사이를 비식별 관계(점선)로 연결합니다.
툴에서 해당 관계선(점선)을 더블클릭하여 관계 속성 창을 엽니다.
속성 창 메뉴 중 [참조무결성] 또는 [Delete Rule (삭제 규칙)] 탭을 찾습니다.
부모 삭제 시(Parent Delete) 규칙을 기본값인 Restrict(제한)나 No Action에서 Cascade(연쇄 삭제)로 변경해 줍니다.

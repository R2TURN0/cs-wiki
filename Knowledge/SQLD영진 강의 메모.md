## ERD

관계 표현 3요소
- 관계명(membership): 두 엔터티간 논리적인 관계
  - 현재동사
  - 양방향이라 두번써야할수도
  - IE, 바커 표ㅣㄱ법 동일
- 관계차수(degree/cardinality): 인스턴스끼리 얼마나 관계에 참여하는지(1:1, 1:N, N:M 등)
  - IE표기법에선 1 작대기 쓰고 바커는 안씀
- 관계선택성(optionality): 인스턴스가 관게에 반드시 참여해야 하는지
  - IE: O 있으면 필수 아니고 선택(optional)
  - 바커: 반대쪽 선을 점선으로 표시

- 업무기술서, 장표에 관계연결에 대한 규칙이 서술되어있는가?(관계차수, 선택성)

# 식별자

엔터티 안의 인스턴스를 유일하도록 식별할 수 있는 속성의 집합

모든 엔터티는 하나 이상의 식별자를 가져야 함

식별자의 분류
- 대표성 여부: 엔터티를 대표하는가
  - 주식별자(PK): 보조식별자 중 하나
    - 특징
      - 유일성: 인스턴스를 유일하게 식별
      - 최소성: 최소한의 속성 조합으로 구성
      - 불변성: 잘 안변해야함
      - 존재성: 존재해야만 함. null안됨
    - 도출 기준
      - 업무에서 자주 사용될 것
      - 이름, 내역 같이 이름으로 되거나 불명확한것은 배제
      - 너무 많은 속성으로 이뤄지지 X
  - 보조 식별자: 선택되지 않은 나머지
- 스스로 발생 여부: 엔터티에 원래 있던건가
  - 내부 식별자(주로 PK)
  - 외부 식별자(FK)
- 단일 속성 여부: 속성 하나로 구성되었는가
  - 단일 식별자
  - 복합 식별자
- 대체 여부: 업무상 편의를 위해 대체한 것인가(복합이던 아니건 무관)
  - 본질식별자
  - 인조식별자


식별 관계/비식별 관계


식별관계(강한연결관계): FK 가 주식별자의 일부
- IE 표기법: 실선
- 바커 표기법: 1로 표시
- 값 필수
- 데이터 독립 생성 불가능, 부모 필수
- 부모 삭제시 자식도 삭제
- 상속할때마다 받는놈은 주식별자 개수 늘어남(확장)
비식별 관계(약한연결관계): FK가 일반속성
- IE 표기법: 점선
- 바커 표기법: 표시 없음
- 값 선택
- 데이터 독립 생성 가능, 부모 필수 아님
- 부모 삭제시 자식은 남음
- 상속해도 주식별자 개수 유지(차단)

식별만 쓰면 식별자가 너무 많아져 조건이 많아짐

비식별만 쓰면 조인이 너무 증가함

# 정규화

정규화: 이상현상을 제거하는 작업

1, 2, 3, BCNF, 4, 5 NF 가 있다.

이전단계를 먼저 완료해야 다음단계됨

정규형: 정규화 된것

비정규형: 아예 정규화 안된것

## 이상현상

삽입이상: 원치않은 정보가 입력되거나 입력을 할 수 없음

갱신이상: 데이터가 변경되며 일관성이 깨짐

삭제이상: 데이터 삭제할 때 원치않는 정보도 삭제됨

## 정규화

함수적 종속: f(x)=y => y는 x에 함수적으로 종속됨

X : {학생 ID} -> Y : {학생명, 성별, 학과코드, 학과명}

## 1차 정규화

도메인 원자성 확포: 도메인이 다중값이거나 비슷한 속성이 여러개면 하나씩만 둬라

## 2차 정규화

부분종속성 제거: PK의 일부에 종속되는 속성 제거

## 3차 정규화

이행종속성 제거: 일반속성끼리 종속 일어나는거 제거

## BCNF

모든 결정자가 후보키여야 함: 일반속성이 다른 속성을 종속하는거 제거

## 4차 정규화

다치 종속 제거: 독립적인 1:N 관계를 한 테이블에 여러개 담지 말것

## 5차 정규화

조인 종속 제거: 테이블을 더 쪼갰다가 다시 join했을때 데이터 손실 없이 복원된다면 테이블을 쪼개라

## 반정규화

논리적 모델링에서 정규화를 통해 분리된 엔터티를 물리적 모델링에서 조회 성능 향상을 위해 다시 합치는 작업

조회 많이 하는 경우 조인이나 봐야 하는 엔터티가 많을 때 시행

엔터티 통합, 중복 허용

데이터 품질 저하됨

정규화는 때로는 조회성능 증가하지만 보통 감소하고

삽입/삭제/수정 성능은 무조건 증가

반정규화는 조회성능 올라감

삽입/삭제/수정 성능은 무조건 감소

### 반정규화 결정 단계

1. 정규화 수행

2. 조회 성능 분석

3. 대안 검토
- 인덱스, 파티셔닝 등

4. 반정규화 적용

### 반정규화 종류

컬림 반정규화: 컬럼을 중복하는 방식

# 트랜잭션

하나의 논리적인 업무 단위

모두 성공하거나 모두 실패해야함

## AICD

1. Atomic(원자성): 트랜잭션은 모두 성공하거나 모두 실패해야함
- 실행방법: commit, rollback
- 실무에서는 근데 중간에 실패해도 거기까지 저장하는걸 원하기도

2. Consistency(일관성): 트랜잭션(SQL실행)은 실행 전과 후가 
- 실행방법: 제약조건

3. Isolation(고립성): 트랜잭션은 서로 간섭할 수 없음
- 실행방법: 고립 레벨

4. Durability(영속성): 트랜잭션 처리 결과는 commit시 DB에 영구반영 되어야 함
- 실행방법: commit

# NULL

NULL: 데이터 저장 공간은 존재하지만 값은 없는 상태

공집합: 공간 자체가 없는 상태

비교나 산술연산 하면 값 다 NULL됨

집계에서는 제외됨, COUNT(컬럼명)의 경우 null 뺴고 세지만 COUNT(*)의 경우 전부 셈

DUAL: DUMMY라는 속성에 X라는 값 하나만 존재하는 오라클의 테스트용 엔터티

WITH A AS (select 구문들) : ()안에 있는 컬럼들을 갖는 A 테이블 임시 생성

NULL 표현

- IE
  - 알 수 없음
- Barker
  - 속성 표시할 때 * 이면 null 불가인 필수 속성, o면 null 가능한 선택 속성

# 본질식별자/인조식별자

- 본질식별자: 원래 존재하는 속성, 자체로 의미가 있음
  - DB에서 중복 차단, 데이터 무결성 보장, 인덱스 관리 편함
  - 키관리 복잡, 조건 관리 복잡
  - 보통 식별관계
- 인조식별자: 원래는 없지만 편의를 위해 만든 속성, 자체로는 의미가 없음
  - 키관리 편함, 조건 관리 편함
  - 로직으로 중복 처리 필요, 애플리케이션에서 데이터 무결성 확보 필요, 인덱스 낭비
  - 보통 비식별 관계

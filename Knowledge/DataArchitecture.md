## 데이터 모델링
데이터 모델링: 현실 세계의 업무를, 데이터를 담을 테이블 구조로 옮겨 그리는 설계 작업
- Entity(엔티티): 업무가 다루는 대상
  - 보통, 명사이며 하나의 Entity가 하나의 테이블이 됨
  - 판단 기준: 이걸 따로 저장하고 추적할 필요가 있는지
- Attribute(속성): Entity의 세부 정보
  - 보통, 컬럼이 됨
- Relationship(관계): Entity 간의 관계
  - 예시: 회원은 주문을 **여러 건** 한다.

Instance: Entity에 해당하는 개체 한개, Entity 타입이 인간이면 Instance는 문종민

### 모델 설계 3단계
1. 개념 모델
  a. 목적: 무엇을 다룰지 합의
  b. 큰 덩어리만 생각함, 세부적인 내용 무시
2. 논리 모델
  a. 목적: DB에 독립적으로, 업무 규칙을 정확히 표현
  b. 개념을 받아서 속성·관계·규칙을 구체화
  c. 식별자 설정
  d. N:M 관계 풀어내기
4. 물리 모델
  a. 목적: 실제로 돌아가고, 빠른 데이터베이스 만들기
  b. 논리 모델을 특정 DBMS(MySQL 등)에 맞춰 구현
  c. 테이블명/컬럼명 확정, 타입 지정
  d. 인덱스, 제약조건, 성능 고려

### 프로세스 관점과 데이터 관점의 차이
프로세스 관점: 일의 흐름 — "주문 → 결제 → 배송 → 완료"
데이터 관점: 데이터의 생성→변화→저장 — "어떤 데이터가 남는가?"
모델링을 하려면 흐름 속에서 "무엇이 데이터로 남아야 하나" 를 골라내야 한다



## 식별 관계 vs 비식별 관계

### 식별 관계 (Identifying Relationship)
자식 엔티티의 PK(기본키)에 부모의 FK가 포함됩니다.
즉, 자식은 부모의 식별자 없이는 자기 자신을 식별할 수 없는 존재입니다.
예: 주문 (PK: order_id) — 주문상세 (PK: order_id + line_no). 주문상세는 주문 없이는 존재 의미가 없고, PK 자체가 부모 키에 의존.

### 비식별 관계 (Non-identifying Relationship)
자식 엔티티의 FK가 일반 컬럼으로만 존재하고, PK와는 무관합니다.
자식은 자기 고유의 PK(보통 surrogate key)를 따로 가지며, 부모는 그냥 "참조"만 함.
예: 부서 (PK: dept_id) — 직원 (PK: emp_id, FK: dept_id). 직원은 부서가 바뀌어도 여전히 같은 emp_id로 식별 가능.

## DAW1 과제
N:M의 관계면 중간 테이블을 놓아 1:N으로 푸는 것이 좋다.
그러나 쇼핑몰에서 Order 테이블과 Item 테이블 사이에 OrderItem 테이블을 놓을 이유가 있을까?
상품 품목과 가격, 이름 등은 주문 시점 당시 기준으로 기록되어야 한다. Item 테이블의 데이터는 계속 변동되기에 단순히 Item ID와 Order ID만을 연결하는 테이블을 두는 것은 의미가 없다.
결국 주문 당시 시점의 데이터를 기록해두어야 하는데 이를 관리할 테이블을 굳이 따로 빼어야 하는가? 그냥 Order 테이블에 주문 품목 컬럼에 기록해 두는 것이 낫지 않을까?

중간 테이블이 필요할 상황
- 주문 이후 물품 추가/제거 등의 기능을 구현해야 한다면 OrderItem과 같은 장바구니 테이블이 필요할 것으로 보인다
- 판매량 데이터에 대해 자세한 분석이 필요하다면 같은 데이터여도 보다 세부적으로 나눠서 저장해두는 중간 테이블이 있는 것이 유리할 것이다.
중간 테이블이 필요 없는 상황
- 규모가 작은 쇼핑몰이거나 판매량 데이터에 대해 세부적인 분석을 할 계획이 없다면 그냥 주문 상세에 각 품목과 개수를 텍스트로 저장하고, 총 가격을 따로 컬럼으로 저장해도 충분할 것이다.
- 조인을 여러번 할 필요가 없기에 조회 성능 면에서 유리할 것으로 보인다.
- 데이터 분석이 필요하다면, 판매량 데이터 분석 목적에 맞게 따로 데이터 저장용 테이블을 구현하는 것이 유리하지 않을까? 주문 체결이 완료된 후 관련 데이터들을 따로 기록하는 테이블을 둔다면 주문/판매 로직에 영향을 미치지 않고 데이터 분석을 할 수 있을 것이다.

결국 상황과 목적에 따라 설계가 달라질 것 같다.

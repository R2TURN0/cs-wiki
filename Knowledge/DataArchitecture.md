# Data Architecture / 데이터 모델링

데이터 모델링과 ERD 표기법을 학습하며 정리한 문서

---

## 데이터 모델링이란

현실 세계의 업무를, 데이터를 담을 테이블 구조로 옮겨 그리는 설계 작업. 핵심 요소는 Entity, Attribute, Relationship 3가지다.

### Entity(엔티티): 업무에서 관리하거나 처리해야 하는 주요 대상

- 표로 만들어서 저장하고 싶은 것
- 공통된 특성으로 하나로 묶을 수 있는 데이터의 집합
- 보통 명사이며, 하나의 Entity가 하나의 테이블이 됨
- 판단 기준
  - 업무에 필요한가?: 시스템이 꼭 관리해야 할 정보인가
  - 여러 개 모이는가?: 인스턴스가 2개 이상 생기는가
  - 식별이 되는가?: 하나하나를 구별할 이름/번호가 있는가
- 종류: 보통 12가지가 있으며, 대표적인 2가지는 아래와 같다.
  - **개체(키) 엔티티**: 그 자체로 존재하는 대상 (예: 회원, 상품, 사원)
  - **행위(내역) 엔티티**: 무언가 일어난 사건/행위 (예: 주문, 결제, 로그인이력)
  - 개체는 "명사", 행위는 "~했다는 기록"으로 생각하면 쉽다.

### Attribute(속성): 엔티티 안에서 관리하고 싶은 정보 항목 하나하나

- 엔티티의 세부 정보, 본질적인 성질이며 보통 컬럼이 됨
- 핵심 개념
  - **원자값**: 더 쪼개면 의미가 깨지는 가장 작은 단위. 일단 최소 단위로 쪼갠 뒤 필요에 따라 합친다. 너무 쪼개는 것도, 안 쪼개는 것도 문제라서 업무 요구사항에 따라 결정한다.
  - **단일값**: 하나의 인스턴스는 그 속성 값을 딱 하나만 가져야 한다. 반복되는 값은 별도 엔티티로 분리한다(제1정규화). 이 역시 상황에 따라 결정한다.
- 판단 기준
  1. 최소 단위 분할: 원자 단위까지 쪼갰는가? (목적 고려)
  2. 유일값 검증: 값이 하나뿐인가? 반복되면 분리한다.
  3. 가공값 판정: 계산으로 다시 만들 수 있는 값인가? 재현 가능한 값은 논리모델에서 보통 생략한다.
  4. 관리수준 검토: 더 상세히 관리할 필요가 있는가?
- 필수/선택 속성
  - 필수(Mandatory): 값이 반드시 있어야 함 — 표시 `*`
  - 선택(Optional): 값이 없어도 됨 — 표시 `o`

### Relationship(관계): 엔티티 사이의 업무적 연관성

- 예시: 회원은 주문을 **여러 건** 한다.
- 업무 규칙을 담으며, DB에서는 참조 무결성(referential integrity)을 지켜주는 핵심 장치다.
- 코드 레벨에서는 외래키 컬럼으로 구현된다.
- 핵심 개념 3가지
  - 관계명
  - **기수성(Cardinality)**: 한 엔티티의 인스턴스 하나가 상대 쪽 인스턴스를 몇 개 가질 수 있는가. 대부분의 실무 관계는 1:N으로 정리된다.
  - **선택성(Optionality)**: 관계가 반드시 있어야 하는가, 없을 수도 있는가. 기수성과 선택성은 방향마다 따로 본다.
- 두 엔티티에 관계가 있다는 것은, 부모의 식별자(PK)를 자식이 관계속성(FK)으로 물려받는다는 의미다.
  - **식별 관계**: 부모의 식별자(FK)가 자식의 식별자(PK)의 일부가 됨. 까마귀발 실선. 자식이 부모 없이는 식별조차 안 되는 강한 결합일 때만 사용한다.
  - **비식별 관계**: 부모의 식별자(FK)가 자식의 식별자(PK)의 일반 컬럼이 됨. 까마귀발 점선. 대부분의 관계는 비식별 관계다.
- 관계의 종류
  - **1:1 관계**: 자주 발생한다면 두 엔티티가 사실 같은 Entity일 가능성이 높다.
  - **1:N 관계**: 가장 흔하다. 수평보다는 수직적 관계가 많다.
  - **N:M 관계**: 아직 덜 풀린 상태. 관계형 DB는 FK 하나로 양쪽을 다 가리킬 수 없어 M:N을 직접 표현할 수 없으므로, **교차 엔티티(중간 테이블)**로 풀어줘야 한다.

#### 교차 엔티티가 필요할 때

- 관계 자체에 속성을 채워야 할 때 (수량, 가격, 신청일 등)
- 관계가 또 다른 자식 엔티티를 가져야 할 때
- 관계명을 못 붙이겠을 때

#### 다중 관계 해결법

- **병렬식**: FK 컬럼을 여러 개 늘린다. 종류가 고정·소수일 때 유용하다.
- **직렬식**: 관계 엔티티를 따로 둔다. 종류가 늘어나거나 이력이 필요할 때 유용하다.

### Instance

Entity에 해당하는 개체 한 개. Entity 타입이 "인간"이면 Instance는 "문종민". Entity는 Instance의 집합이다.

### SubType

- **슈퍼타입**: 공통 속성·관계를 보유
- **서브타입**: 자기만의 속성을 추가로 가짐
- 규칙: 서브타입끼리 겹치지 않고, 다 합치면 슈퍼타입과 1:1이 된다.
- 예시: 개인회원/법인회원(서브타입)을 합치면 회원(슈퍼타입)이 된다.
- 특징
  - 모든 회원은 반드시 하나의 서브타입에 속한다.
  - 애매해서 다 못 나누겠다면 '기타' 구분을 만든다.
  - 슈퍼/서브는 부모-자식 관계가 아니라, 같은 대상을 보는 관점 차이일 뿐이다.
  - 차이가 거의 없다면 굳이 나누지 말고 그냥 속성 하나로 처리한다.

## 모델 설계 3단계

1. **개념 모델**: 무엇을 다룰지 합의하는 단계. 큰 덩어리만 생각하고 세부 내용은 무시한다.
2. **논리 모델**: DB에 독립적으로 업무 규칙을 정확히 표현하는 단계. 개념을 받아 속성·관계·규칙을 구체화하고, 식별자를 설정하며, N:M 관계를 풀어낸다.
3. **물리 모델**: 실제로 돌아가고 빠른 데이터베이스를 만드는 단계. 논리 모델을 특정 DBMS(MySQL 등)에 맞춰 구현하며, 테이블명/컬럼명을 확정하고 타입을 지정한다. 인덱스, 제약조건, 성능을 고려한다.

> [!TIP]
> 항상 데이터 시뮬레이션을 해볼 것.

### 프로세스 관점과 데이터 관점의 차이

- **프로세스 관점**: 일의 흐름 — "주문 → 결제 → 배송 → 완료"
- **데이터 관점**: 데이터의 생성 → 변화 → 저장 — "어떤 데이터가 남는가?"

모델링을 하려면 흐름 속에서 "무엇이 데이터로 남아야 하는가"를 골라내야 한다.

## 식별 관계 vs 비식별 관계

### 식별 관계 (Identifying Relationship)

자식 엔티티의 PK(기본키)에 부모의 FK가 포함된다. 즉, 자식은 부모의 식별자 없이는 자기 자신을 식별할 수 없는 존재다.

- 예: 주문(PK: `order_id`) — 주문상세(PK: `order_id + line_no`). 주문상세는 주문 없이는 존재 의미가 없고, PK 자체가 부모 키에 의존한다.

### 비식별 관계 (Non-identifying Relationship)

자식 엔티티의 FK가 일반 컬럼으로만 존재하고 PK와는 무관하다. 자식은 자기 고유의 PK(보통 surrogate key)를 따로 가지며, 부모는 그냥 "참조"만 한다.

- 예: 부서(PK: `dept_id`) — 직원(PK: `emp_id`, FK: `dept_id`). 직원은 부서가 바뀌어도 여전히 같은 `emp_id`로 식별 가능하다.

## 까마귀발 표기법

| 위치 | 기호 | 의미 |
| --- | --- | --- |
| 바깥쪽 | `\|` | 하나(One) |
| 바깥쪽 | `<` | 여러 개(Many) |
| 안쪽 | `o` | 선택 (0개 가능, optional) |
| 안쪽 | `\|` | 필수 (1개 이상, mandatory) |

## DAW1 과제 회고: 중간 테이블은 언제 필요한가

N:M 관계면 중간 테이블을 놓아 1:N으로 푸는 것이 일반적으로 좋다. 그러나 쇼핑몰에서 Order 테이블과 Item 테이블 사이에 OrderItem 테이블을 놓을 이유가 있을까?

상품의 품목, 가격, 이름 등은 주문 시점 당시 기준으로 기록되어야 한다. Item 테이블의 데이터는 계속 변동되기 때문에, 단순히 Item ID와 Order ID만 연결하는 테이블을 두는 것은 의미가 없다. 결국 주문 당시 시점의 데이터를 기록해두어야 하는데, 이를 관리할 테이블을 굳이 따로 빼야 하는가? 그냥 Order 테이블에 주문 품목 컬럼으로 기록해 두는 것이 낫지 않을까?

**중간 테이블이 필요한 상황**

- 주문 이후 물품 추가/제거 등의 기능을 구현해야 한다면, OrderItem과 같은 장바구니 테이블이 필요할 것으로 보인다.
- 판매량 데이터에 대해 자세한 분석이 필요하다면, 같은 데이터여도 더 세부적으로 나눠서 저장해두는 중간 테이블이 유리하다.

**중간 테이블이 필요 없는 상황**

- 규모가 작은 쇼핑몰이거나 판매량 데이터에 대한 세부 분석 계획이 없다면, 주문 상세에 각 품목과 개수를 텍스트로 저장하고 총 가격만 따로 컬럼으로 저장해도 충분할 것이다.
- 조인을 여러 번 할 필요가 없어 조회 성능 면에서 유리하다.
- 데이터 분석이 필요하다면, 분석 목적에 맞게 별도의 데이터 저장용 테이블을 두는 편이 유리할 수 있다. 주문 체결이 완료된 후 관련 데이터를 따로 기록하는 테이블을 두면, 주문/판매 로직에 영향을 주지 않고 데이터 분석을 할 수 있다.

결국 상황과 목적에 따라 설계가 달라진다.

## 스냅샷과 반정규화 설계

**상황**: 주소 테이블의 일반 속성인 주소 항목만, 주문 테이블의 일반 필수 속성으로 그대로 받고 싶은 경우.

이런 설계는 다음과 같이 부른다.

- **스냅샷(Snapshot) 데이터**: 특정 시점(주문 시점)의 상태를 사진 찍듯 그대로 저장해두고, 원본(주소 테이블)이 변경되더라도 영향을 받지 않도록 하는 설계.
- **속성의 반정규화(Denormalization) / 데이터 중복**: 정규화 원칙대로라면 '주소ID(FK)'만 가져와서 조인해야 하지만, 과거 데이터 보존과 조회 성능을 위해 의도적으로 '주소 텍스트' 자체를 주문 테이블에 중복으로 복사해두는 방식.
- **관계 단절(Decoupling)**: 테이블 간 외래키(FK) 참조나 논리적 연결 고리를 고의로 끊어버린 상태.

표현 방법

1. **관계선을 그리지 않는다.** 바커 표기법을 포함한 ERD에서 두 엔티티 사이에 선을 긋는다는 것은 '참조 관계(연동)'가 성립해 외래키(FK)로 연결됨을 의미한다. 시점 데이터만 복사하고 연동하지 않을 것이므로, 주소 엔티티와 주문 엔티티 사이에는 아무 선도 연결하지 않는 것이 맞다.
2. **복사해 온 주소는 `*`(필수 속성)로 표기한다.** 주문 엔티티 안에 '주소' 속성을 추가하고, 바커 표기법의 필수 기호인 `*`(asterisk)를 붙인다.

> [!NOTE]
> 요약: 관계는 표시하지 않는다.

## 첸(Chen) 표기법

- **이중 사각형**: 약한 엔티티를 의미한다.
- **이중선(전체 참여, Total Participation)**: "대여가 존재하려면 반드시 회원이 있어야 하고, 반드시 킥보드가 있어야 한다"처럼 필수적인 관계에는 선을 이중선(두 줄)으로 긋는다. 예를 들어 대여 상자에서 '신청하다'와 '점유하다'로 가는 선을 두 줄로 그으면, "대여 데이터는 회원과 킥보드 없이 독립적으로 존재할 수 없다"는 비즈니스 규칙을 시각화할 수 있다.
- **유도 속성(점선 동그라미, Derived Attribute)**: 대여 엔티티의 이동 거리나 이용 시간처럼, '시작 시간'과 '종료 시간'으로 계산해낼 수 있는 값. Chen 표기법에서는 점선 타원으로 그리기도 한다.

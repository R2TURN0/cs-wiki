# Big Data

빅데이터와 데이터 프로덕트 개발 과정에서 배운 내용을 정리하는 문서

---

## 빅데이터의 정의

**High volume, velocity, variety**를 가진 정보. 데이터 접근성(accessibility)이 핵심이다.

- 코드가 있고 그 자리에 데이터를 두는 것이 아니라, **데이터가 있는 곳으로 코드를 보내는 것**이 원칙이다 (data locality).
- 이를 위해 **data parallelism**(데이터를 나눠 병렬로 처리하는 방식)을 사용한다.

## 처리 성능 관련 원칙

- **셔플(shuffle)을 막는 것**이 중요하다. 셔플은 노드 간 데이터 이동을 유발해 성능을 크게 떨어뜨린다.
- Extract 단계는 CPU 코어 수를 늘려도 큰 의미가 없다.
- Transform 단계는 CPU 성능, 코어 수, 메모리가 좋을수록 유리하다.

## Dockerfile 작성 원칙

- Dockerfile은 연습장이 아니다. 제출 시에는 다른 사람이 재사용할 수 있도록 정리해서 작성한다.
- 시도했던 과정을 전부 남기지 말고, 필요한 내용만 담는다.

## 작업 방식

- 한 줄씩 직접 실행해보고 바로바로 피드백을 받는 방법을 찾는다.
- **타임박싱(timeboxing)**이 중요하다. "30분 동안만 이걸 하자"처럼 제한 시간을 두고, 시간이 다 되면 미완성이어도 다음 단계로 넘어간다.
- 어떤 데이터가 있었다면 어떤 상황에서 어떤 다른 선택을 할 수 있었을지 스스로 물어본다.

## 프로토타이핑

- 프로토타이핑은 가라 데이터(임시 데이터)를 챗GPT 등으로 만들어 차트를 뽑아 보여주는 수준으로 충분하다. 굳이 데이터 파이프라인까지 구축할 필요는 없다.
- 목적은 방향이 맞는지 확인하고 의사결정을 돕는 것이므로, 대강 빠르게 만들어 "이걸 진짜 만드는 게 맞을지"를 확인하는 데 집중한다.
- **프로토타이핑 ≠ Data Product Development.** 프로토타이핑은 Data Product Development를 애자일하게 진행하는 방법 중 하나일 뿐이다.
- **EDA ≠ 프로토타이핑.** EDA에는 상품에 대한 개념이 없다.

## 프로덕트 개발을 잘 하려면

- **Use case를 아주 구체적으로 잡는다.**
  - 고통이 큰 문제를 정의한다.
  - 그 문제를 푸는 use case를 구체적으로 좁힌다.
- 주제와 고객을 좁게 정의한다.
- 비싸게 팔 수 있다는 가정이 없다면 시작하지 않는다 (예: B2B 500만 원 이상).
- 비용을 줄이는 것보다 매출을 높이는 방향이 더 매력적이다.

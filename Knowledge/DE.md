# Data Engineering

하둡/YARN 등 데이터 엔지니어링 기초 개념과 학습 태도를 정리하는 문서

## 학습 태도

- 생각 없이 따라 하지 않는다.
- **타임박싱**: 20분간 뭘 하기로 했으면 시간이 딱 되었을 때 멈추고, 다시 20분을 더 할지 동료에게 피드백을 받을지 등을 판단한다.
- 배가 산으로 가지 않도록 작업을 잘게 쪼개서 진행한다.
- 자기만의 그라운드 룰을 설정한다.
- 시킨 대로만 하지 말고 왜 시키는지 이해하고 한다. (겉으로 드러나지 않는 문제도 숨어 있을 수 있다.)

## 하둡(Hadoop)

거대한 데이터를 저렴한 하드웨어에서 효율적으로 저장·처리하기 위해 등장했다.

- **공통 레이어**: HDFS, Hadoop YARN
- 이 공통 레이어 위에 **맵리듀스**가 올라가면 하둡, **스파크**가 올라가면 스파크가 된다. 공통 레이어 자체도 다른 엔진으로 대체 가능하다.
- 마스터 노드에서는 맵리듀스 코드가 직접 돌지 않고, 슬레이브 노드에서만 실행된다. 마스터는 기록과 작업 분배를 담당한다.

### 병렬 처리 방식

- **Data parallelism**: 데이터가 있는 곳에서 작업한다. 즉, 데이터가 있는 곳으로 코드를 보낸다.
- **Task parallelism**: 데이터를 읽어와서 작업한다. 즉, 코드가 있는 곳으로 데이터를 보낸다.

### HDFS의 전제

- **Hardware failure를 가정**하고 replication(복제)으로 해결하며, replication은 우선순위가 높다.
- Write는 동시에 할 수 없어 read보다 느리다.
- **WORM(Write Once Read Many)**: 최신 HDFS는 지원하지만, 우리가 쓰는 구버전은 지원하지 않는다.
  - 로그처럼 잘 바뀌지 않는 데이터에 적합하다. 타임스탬프가 붙는 순간 사실상 로그로 볼 수 있다.
  - 수정이 빈번한 RDB용 데이터는 하둡에 넣지 않는다.

### 노드 구성

- **Namenode**: 파일시스템 메타데이터를 저장한다. 실제 데이터는 절대 오가지 않는다.
- **Datanode**: 파일시스템의 실제 읽기/쓰기를 담당한다.
- Replication도 데이터노드끼리 통신해서 처리하며, 네임노드는 명령만 내린다.
- 같은 랙(rack)에 속한 노드끼리는 같은 스위칭 허브를 공유한다.

### 데이터 지역성(Locality)

데이터가 있는 곳에서 코드를 돌리지 못할 경우, 아래 순서로 대안을 찾는다. 데이터를 최대한 적게 옮기는 것이 핵심이며, 데이터가 네트워크를 타면 성능이 크게 떨어진다.

1. **데이터 로컬**: 같은 노드에 데이터와 코드가 함께 있음
2. **랙 로컬**: 같은 랙의 다른 노드에 데이터, 다른 노드에 코드
3. **다른 랙**: 다른 랙의 노드에 데이터

## YARN

| 구성 요소 | 위치 | 역할 |
| --- | --- | --- |
| RM (Resource Manager) | 클러스터당 1개, 마스터 노드에서 실행 | AM에게 어떤 노드의 컨테이너를 쓸지 스케줄링만 한다 (모니터링은 하지 않음). 권한이 가장 세다. |
| AM (Application Master) | 애플리케이션당 1개, 노드 중 하나에서 실행 | RM으로부터 할당받은 컨테이너를 사용하며, application failure를 관장한다. |
| NM (Node Manager) | 노드당 1개 | 하드웨어를 관리하며 hardware failure를 관장한다. |
| Application Manager | 마스터 노드에서 실행 | job 유효성 체크, application 실행을 위한 AM 지정, AM이 죽으면 재시작한다. |

- k, v 등 중간 데이터를 저장할 때는 HDFS 대신 로컬 디스크를 쓴다. HDFS는 느리기 때문이다. 로컬 디스크 대신 메모리를 쓰면 스파크가 된다.
- 읽어올 데이터와 최종 데이터만 HDFS에 저장한다. 이 데이터로 이후 비즈니스 인사이트를 뽑아낼 수 있다.

### 용어

- **스플릿(Split)**: 고정된 크기로 나눈 데이터
- **파티션(Partition)**: 단일 키나 몇 개의 키로 모아둔 데이터

### 파티션 편향(Skewing)

특정 파티션에 데이터가 몰리는 skewing 문제는, 옛날 하둡에서는 큰 파티션에 리듀서를 여러 개 붙이는 방식을 잘 쓰지 않았지만 스파크에서는 사용한다. 데이터를 직접 돌려보며 튜닝이 필요하다.

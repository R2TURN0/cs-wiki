# 데이터 아키텍트 기본 지식 리뷰 (질의응답 20분 대비)

> 공유 자전거 서비스 프로젝트(고객/자전거/대여/결제/정비) 사례에 맞춰 정리

---

## 1. 정규화 / 반정규화

### 정규화 단계 요약

| 단계 | 조건 | 위반 예시 (일부러 만든 나쁜 설계) |
|---|---|---|
| 1NF | 모든 속성이 원자값(하나의 값만 가짐) | `TB_BICYCLE.OWNER_TAG`에 "1호점,2호점"처럼 콤마로 여러 값을 넣는 것 |
| 2NF | 부분함수종속 제거 (복합PK의 일부에만 종속되는 속성 없음) | 결제상세 테이블 PK를 (결제ID, 상품ID)로 잡았는데, 결제일시가 결제ID에만 종속되면서도 이 테이블에 반복 저장 |
| 3NF | 이행함수종속 제거 (비키 속성이 다른 비키 속성에 종속 X) | `TB_RENTAL`에 자전거모델명까지 저장 (자전거ID → 자전거모델ID → 모델명 이라는 이행 종속) |
| BCNF | 모든 결정자가 후보키 | 실무에서 3NF까지면 충분한 경우가 대부분 |

**우리 프로젝트에서 3NF를 지킨 예:** `TB_RENTAL`은 `BICYCLE_ID`만 갖고 `BICYCLE_MODEL_ID`나 모델명을 직접 갖지 않음 → 자전거 모델이 바뀌어도 과거 대여기록을 수정할 필요 없음.

**의도적으로 정규화를 깬 곳(반정규화) — 왜 예외인지 설명 가능해야 함:**
- `TB_RENTAL.RETURN_POLICY_ZONE_ID`, `RETURN_SERVICE_REGION_ID` : 반납 시점 스냅샷 (계산 비용 절감 + 이력 보존 목적)
- `TB_PAYMENT`의 계산 중간값들(잠금해제금액, 차감금액, 실결제금액) : 감사추적(audit trail) 목적의 스냅샷

> **질문 대비**: "정규화를 왜 안 지켰나?" → "정규화 위반이 아니라, 이력/스냅샷 목적의 의도적 반정규화이며 원본 마스터(요금정책 등)가 바뀌어도 과거 기록은 훼손되지 않아야 하기 때문"이라고 답하면 됩니다.

### 4NF, 5NF — 실무에선 드물지만 개념은 알아두기

3NF/BCNF까지는 "함수적 종속(X → Y, 하나의 값이 다른 하나의 값을 결정)"을 다뤘습니다. 4NF, 5NF는 **함수적 종속이 아니라 "다치 종속"과 "조인 종속"이라는, 조금 다른 성격의 중복 문제**를 다룹니다.

**4NF (제4정규형) — 다치종속(Multi-valued Dependency) 제거**

다치종속이란: "X 하나의 값에 대해, Y값들의 집합이 정해지는데, 그 집합이 **다른 독립적인 속성 Z와는 무관하게** 결정되는 경우"를 말합니다. 문제는 서로 **관련 없는 두 다치종속을 한 테이블에 같이 넣으면** 불필요한 조합이 다 생겨서 중복이 발생한다는 거예요.

예시로 설명하면 (프로젝트에 실제로는 없는, 이해를 위한 가상 사례): 만약 "직원 한 명이 여러 자격증을 갖고, 동시에 여러 근무가능 요일을 갖는다"를 이렇게 한 테이블에 넣었다고 해봅시다.

| 직원ID | 자격증 | 근무가능요일 |
|---|---|---|
| E1 | 자전거정비1급 | 월 |
| E1 | 자전거정비1급 | 수 |
| E1 | 전기안전자격 | 월 |
| E1 | 전기안전자격 | 수 |

직원E1이 자격증 2개, 근무요일 2개를 가진다면, **자격증과 근무요일은 서로 아무 상관이 없는데도** 모든 조합(2×2=4행)을 다 만들어야 저장이 됩니다. 이게 다치종속 위반이에요. 해결책은 **자격증 테이블과 근무요일 테이블을 완전히 별개로 분리**하는 것 (4NF를 만족시키려면 서로 독립적인 다치종속은 각각 별도 테이블로 분리).

> 우리 프로젝트에서 비슷한 패턴이 생길 수 있는 지점: 만약 "직원이 여러 자격증을 갖고, 동시에 여러 근무시설에 배정된다"는 요구사항이 생긴다면 `TB_EMPLOYEE_CERT`(직원-자격증)와 `TB_EMPLOYEE_FACILITY`(직원-근무시설)를 각각 별도 교차 테이블로 분리해야 4NF를 만족합니다. 이 프로젝트는 서브타입 처리와 FK 분리가 꼼꼼해서 이런 다치종속이 남아있는 지점은 없어 보입니다.

**5NF (제5정규형, PJ/NF) — 조인종속(Join Dependency) 제거**

한 테이블을 세 개 이상의 테이블로 쪼갰다가 다시 조인했을 때, **원래 정보와 정확히 똑같이 복원되는가**를 다루는 정규형입니다. 즉 "이 테이블은 더 이상 손실 없이 쪼갤 수 없는 최소 단위인가"를 검증하는 최종 단계예요.

전형적인 예: "공급업체-제품-지역" 3자 관계에서, 실제로는 "이 공급업체가 이 제품을 취급한다", "이 제품이 이 지역에서 팔린다", "이 공급업체가 이 지역에서 활동한다"는 **세 개의 독립적인 이진 관계**일 뿐인데, 이걸 하나의 3원 관계 테이블로 뭉쳐놓으면 실제로 존재하지 않는 조합까지 유추되는 이상 현상이 생깁니다. 이 경우 3원 관계 테이블을 이진 관계 테이블 세 개로 쪼개는 게 5NF입니다.

> 이 프로젝트에 억지로 적용해보면: 만약 "쿠폰-서비스권역-멤버십등급" 3자가 동시에 얽힌 프로모션 규칙이 생긴다면 5NF까지 고려 대상이 될 수 있지만, 현재 구조엔 3원 관계 자체가 없어서 해당사항이 없습니다.

> **질문 대비**: "4NF, 5NF까지 고려했나?" → "이 도메인은 서로 독립적인 다치종속이나 3원 조인종속이 발생할 만한 구조가 아니라서(예: 직원-자격증-근무요일처럼 서로 무관한 다중값 조합이 없음), 실질적으로 3NF 수준에서 이상현상 없이 설계가 완결됩니다. 다만 향후 직원 자격증 관리처럼 다대다 속성이 늘어나면 4NF 원칙에 따라 별도 교차 테이블로 분리할 계획"이라고 답하면 이론적 이해와 실무 판단을 동시에 보여줄 수 있습니다.

## 2. 엔터티(Entity)의 종류

논리 모델링에서 엔터티를 분류하는 기준은 여러 가지가 있지만, 가장 흔히 쓰이는 두 가지 분류축을 알아두면 됩니다.

### (1) 발생 시점/성격에 따른 분류 (기본/중심/행위 엔터티)

| 분류 | 정의 | 판별 기준 | 프로젝트 예시 |
|---|---|---|---|
| **기본(본질) 엔터티**<br>Fundamental Entity | 다른 엔터티 없이도 독립적으로 존재 가능한 실체 | 다른 엔터티를 참조(FK)하지 않고, 오히려 참조당함(피참조). 시간과 무관하게 안정적으로 존재 | `TB_CUSTOMER`, `TB_BICYCLE` |
| **중심 엔터티**<br>Main Entity | 기본엔터티로부터 발생하며, 그 자체가 업무의 핵심 사건(트랜잭션)이 됨 | 기본엔터티들을 FK로 참조해서 만들어짐. 발생 일시가 핵심 속성 | `TB_RENTAL`, `TB_MAINTENANCE` |
| **행위 엔터티**<br>Active/Action Entity | 중심엔터티가 반복적으로 발생하면서 만들어내는 상세/이력성 실체 | 중심엔터티에 종속되어 여러 번 반복 발생 가능 | `TB_PAYMENT`(대여/멤버십/패널티로부터 반복 발생), `TB_FAILURE_REPORT` |

> **질문 대비 (첫 대화에서 실제로 논의했던 부분)**: "고객·대여·자전거·결제·정비가 모두 같은 급의 엔터티인가?" → 아니요. `고객·자전거`는 기본(본질) 엔터티이고, `대여·정비`는 중심 엔터티, `결제`는 행위 엔터티에 가깝습니다. 발표자료의 "5대 영역"은 엔터티 유형 분류가 아니라 **업무기능(주제영역) 단위 분류**라서 기준이 다릅니다 (자세한 논리는 이 문서 앞부분 대화 맥락 참고).

### (2) 존재 형식에 따른 분류

| 분류 | 정의 | 프로젝트 예시 |
|---|---|---|
| **강한 개체(Strong Entity)** | 자신만의 PK로 독립적으로 식별 가능 | `TB_CUSTOMER`, `TB_RENTAL` |
| **약한 개체(Weak Entity)** | 부모 엔터티의 PK 없이는 식별 불가 (식별관계로 연결) | `TB_BICYCLE_LOCATION` (부모 `TB_BICYCLE`의 PK를 그대로 PK로 사용) |

## 3. 속성(Attribute)의 종류

속성도 여러 기준으로 분류됩니다. 논리 모델링 질문에서 자주 나오는 4가지 축을 정리합니다.

### (1) 특성(성질)에 따른 분류

| 분류 | 정의 | 프로젝트 예시 |
|---|---|---|
| **기본속성(Basic Attribute)** | 업무상 원래부터 존재하는, 다른 속성으로부터 계산되지 않는 값 | `CUST_NM`(고객명), `RENT_START_DTTM`(대여시작일시) |
| **설계속성(Designed Attribute)** | 원래 업무엔 없었지만 설계 과정에서 시스템적으로 필요해 추가한 값 | `CUST_ID`(대리키), `CUST_PUBLIC_ID`(UUID) — 업무담당자는 "고객ID"라는 개념 없이 그냥 "그 고객"이라고만 생각하지만, 시스템 구현을 위해 인위적으로 부여 |
| **파생속성(Derived Attribute)** | 다른 속성의 값을 계산하거나 특정 시점의 값을 복사해서 만든 값 | `TB_RENTAL.RETURN_POLICY_ZONE_ID`(반납시점 스냅샷), `TB_PAYMENT`의 계산 중간값들 |

> **질문 대비**: "파생속성을 왜 저장하나, 그냥 조회 시점에 계산하면 안 되나?" → 이미 앞서 다룬 반정규화/스냅샷 논리와 동일. 계산 비용 절감 + 원본이 바뀌어도 과거 값이 훼손되지 않는 이력 보존 목적.

### (2) 분해 가능 여부에 따른 분류

| 분류 | 정의 | 프로젝트 예시 |
|---|---|---|
| **단일값(원자)속성** | 더 이상 의미 있게 쪼갤 수 없는 값 | `BATTERY_LEVEL`(배터리 잔량) |
| **복합속성(Composite Attribute)** | 여러 하위 속성으로 의미 있게 분해 가능한 값 | 주소 = (우편번호 + 상세주소)처럼 논리적으로는 하나지만 내부에 구성요소가 있는 값. `TB_FACILITY`의 주소류 속성이 이에 해당할 수 있음 |

### (3) 값 개수에 따른 분류

| 분류 | 정의 | 프로젝트 예시 |
|---|---|---|
| **단일값속성(Single-valued)** | 하나의 인스턴스가 값을 하나만 가짐 | `CUST_NM` |
| **다중값속성(Multi-valued)** | 하나의 인스턴스가 여러 값을 가질 수 있음 | (1NF 위반 예시로 앞서 든) 자격증, 근무가능요일처럼 여러 개 가질 수 있는 값. **정규화 원칙상 다중값속성은 그대로 저장하면 안 되고, 별도 테이블(자식 엔터티)로 분리해야 함** |

> **질문 대비**: "다중값속성이 있으면 어떻게 처리하나?" → 컬럼에 배열이나 콤마구분 문자열로 넣지 않고, 반드시 별도의 자식 테이블(1:N 관계)로 분리해서 1NF를 만족시켜야 함 (4NF 설명에서 다룬 직원-자격증 예시와 동일한 원리).

### (4) 역할(Role)에 따른 분류

| 분류 | 정의 | 프로젝트 예시 |
|---|---|---|
| **주식별자(PK 속성)** | 엔터티를 유일하게 식별하는 속성 | `RENT_ID` |
| **외래식별자(FK 속성)** | 다른 엔터티를 참조하는 속성 | `TB_RENTAL.CUST_ID` |
| **일반속성(Descriptive Attribute)** | 식별과 무관한, 그 엔터티의 특성을 서술하는 속성 | `RENT_START_COORD`(대여시작좌표) |

## 4. 키(Key)의 종류

| 키 종류 | 정의 | 프로젝트 예시 |
|---|---|---|
| 후보키(Candidate Key) | 행을 유일하게 식별할 수 있는 최소 속성 집합 | 고객의 `CUST_ID`, `CUST_EMAIL` 둘 다 후보키 |
| 기본키(PK) | 후보키 중 대표로 선택된 것 | `CUST_ID` (bigint) |
| 대체키(Alternate Key) | 후보키 중 PK로 뽑히지 않은 나머지 | `CUST_EMAIL`, `PHONE_NO` |
| 보조키(Secondary Key) | 외부 노출용 별도 식별자 (본 프로젝트 고유 용어 사용) | `CUST_PUBLIC_ID` (UUID) |
| 외래키(FK) | 다른 테이블의 PK를 참조하는 속성 | `TB_RENTAL.CUST_ID` |
| 슈퍼키(Super Key) | 유일성만 만족(최소성 불필요), 후보키를 포함하는 상위 개념 | `CUST_ID` 단독도, `(CUST_ID, CUST_NM)`도 전부 슈퍼키 |

> **질문 대비**: "왜 PK는 bigint, 외부노출은 UUID로 이원화했나?" → 조회 성능(정수 비교가 빠름, 인덱스 크기 작음) + 보안(순차값 추측/열거 공격 방지)의 두 마리 토끼를 잡기 위함.

## 5. 관계 표현 — 식별/비식별 관계, 카디널리티

- **식별 관계(Identifying Relationship)**: 부모의 PK가 자식의 PK 일부/전부가 됨 → 부모 없이 자식 존재 불가. 실선.
  - 예: `TB_BICYCLE_LOCATION.BICYCLE_ID`가 PK+FK. 자전거 없이 위치 데이터 단독 존재 불가.
  - 예: 결제 서브타입(카드결제 등)도 부모 결제ID를 그대로 PK로 씀.
- **비식별 관계(Non-identifying Relationship)**: FK가 그냥 일반 속성일 뿐 PK의 일부가 아님 → 자식이 독립적 식별자를 가짐. 점선(도구에 따라 표기 다름).
  - 예: `TB_RENTAL.CUST_ID` — 대여는 자체 `RENT_ID`가 있고, 고객ID는 그냥 참조.
- **카디널리티(1:1, 1:N, N:M)**
  - 1:1 → `TB_BICYCLE : TB_BICYCLE_LOCATION` (자전거 1대당 현재위치 1개)
  - 1:N → `TB_CUSTOMER : TB_RENTAL` (고객 1명이 대여 여러 건)
  - N:M → 원래는 중간 테이블(연결/교차 엔터티)로 풀어야 함. 예를 들어 "쿠폰이 여러 서비스권역에 적용되고, 서비스권역도 여러 쿠폰을 가질 수 있다"면 `TB_COUPON_SERVICE_REGION` 같은 교차 테이블이 필요.

> **질문 대비**: "N:M 관계가 실제 스키마에 있나?" 있다면 교차 엔터티로 어떻게 풀었는지 짚을 수 있어야 하고, 없다면 왜 없는지(이 도메인엔 다대다가 실제로 없다는 점) 설명할 수 있어야 합니다.

## 6. 서브타입(슈퍼타입-서브타입) 처리 전략

일반적으로 3가지 구현 방식이 있습니다 (이미 발표자료에 언급됐지만 이름을 정확히 알아두면 좋음):

| 전략 | 설명 | 장점 | 단점 | 프로젝트 적용 |
|---|---|---|---|---|
| **Single Table Inheritance (단일테이블)** | 슈퍼+서브 속성을 한 테이블에 다 넣고 nullable 처리 | 조인 없음, 단순 | 컬럼 낭비, 어떤 서브타입인지 구분자(discriminator) 필요 | `TB_PAYMENT`가 이 방식 (Roll-up) — `PAYMENT_TYPE_CD`로 구분 |
| **Class Table Inheritance (서브타입별 테이블)** | 슈퍼 테이블 + 서브타입별 테이블을 1:1 식별관계로 분리 | 각 서브타입 속성이 명확, 컬럼 낭비 없음 | 조회 시 조인 필요 | 쿠폰(금액할인/비율할인), 패널티(벌금/정지) 방식 |
| **Concrete Table Inheritance (서브타입 테이블만)** | 슈퍼타입 테이블 없이 서브타입 테이블마다 공통속성 중복 저장 | 조인 불필요 | 공통속성 중복, 슈퍼타입 전체 조회 어려움 | 사용 안 함 |

> **질문 대비**: "왜 결제는 Roll-up, 쿠폰/패널티는 분리했나?" → 결제는 서브타입 간 속성 차이가 거의 없고(대상 FK만 다름) 항상 함께 조회(결제내역 리스트)되는 반면, 쿠폰/패널티는 서브타입마다 **고유 속성이 뚜렷하고(할인율 vs 할인금액, 벌금액 vs 정지기간)** 중복 배타성/포함성 규칙이 있어 별도 테이블이 더 명확하기 때문.

> **추가로 알아둘 개념 — 배타적 서브타입(Exclusive) vs 포함적 서브타입(Inclusive/Overlapping)**
> - 배타적: 한 인스턴스가 여러 서브타입에 동시에 속할 수 없음 (쿠폰: 금액할인 아니면 비율할인, 둘 다는 안 됨)
> - 포함적: 한 인스턴스가 여러 서브타입에 동시에 속할 수 있음 (패널티: 벌금형이면서 동시에 계정정지형일 수 있음)
> - **DB 레벨에서 배타성을 강제로 검증하는 제약은 기본적으로 없고**, 애플리케이션 로직이나 CHECK 제약/트리거로 별도 구현해야 함 (이 부분은 앞서 논의했던 대로 질문받을 가능성 높음)

## 7. 트랜잭션과 ACID

| 속성 | 의미 | 프로젝트 예시 |
|---|---|---|
| **Atomicity(원자성)** | 트랜잭션은 전부 성공하거나 전부 실패 | 대여 시작 시 "자전거 상태를 대여중으로 변경" + "대여 레코드 생성"이 하나의 트랜잭션. 결제 실패 시 둘 다 롤백 |
| **Consistency(일관성)** | 트랜잭션 전후로 DB의 제약조건이 항상 유지됨 | 자전거는 반드시 하나의 상태값(대여가능/대여중/정비중)만 가져야 함 |
| **Isolation(격리성)** | 동시 실행 트랜잭션이 서로 간섭하지 않음 | 두 사용자가 동시에 같은 자전거 대여 시도 시 하나만 성공해야 함 |
| **Durability(지속성)** | 커밋된 데이터는 장애가 나도 유지됨 | 결제 승인 완료 후 서버가 죽어도 결제 기록은 남아있어야 함 |

> **질문 대비**: "동시에 같은 자전거를 두 명이 대여 요청하면?" → `SELECT ... FOR UPDATE`로 해당 자전거 행에 배타락을 걸고, 상태값이 '대여가능'인지 확인 후 변경하는 방식(비관적 락)으로 방어. (아래 8번 항목과 연결)

## 8. 동시성 제어

### 격리수준 (앞서 다룬 내용 요약)

| 수준 | Dirty Read | Non-repeatable Read | Phantom Read | PostgreSQL 기본값 |
|---|---|---|---|---|
| READ UNCOMMITTED | 발생가능 | 발생가능 | 발생가능 | (PG는 사실상 미지원, READ COMMITTED로 동작) |
| READ COMMITTED | 방지 | 발생가능 | 발생가능 | ✅ 기본값 |
| REPEATABLE READ | 방지 | 방지 | 방지(PG는 표준보다 강함) | |
| SERIALIZABLE | 방지 | 방지 | 방지 | 가장 엄격, 재시도 로직 필요 |

### 4단계 격리수준 각각의 의미

**1. READ UNCOMMITTED (커밋 안 된 것도 읽기 허용) — 가장 느슨함**

다른 트랜잭션이 아직 커밋하지 않은 값까지 그대로 읽을 수 있게 허용하는 수준. 세 가지 이상현상(더티리드/반복불가읽기/팬텀리드)이 **전부 발생 가능**합니다. 이론상 가장 빠르지만(락을 거의 안 기다림) 정합성 보장이 없어 실무에서 거의 쓰이지 않습니다.

> **PostgreSQL의 특이점**: `SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED`라고 지정해도, PostgreSQL은 내부적으로 이 레벨을 별도 구현하지 않고 **READ COMMITTED와 동일하게 동작**합니다. 이는 PostgreSQL의 MVCC(다중버전 동시성 제어) 구조상 "커밋 안 된 값을 아예 안 보이게" 만드는 게 기본 설계라, 더티리드 자체가 애초에 발생할 수 없기 때문입니다. 즉 PostgreSQL에서는 4단계 중 사실상 3단계(READ COMMITTED / REPEATABLE READ / SERIALIZABLE)만 의미가 있습니다.

**2. READ COMMITTED (커밋된 것만 읽기) — PostgreSQL 기본값**

다른 트랜잭션이 **커밋을 완료한 값만** 읽을 수 있게 해서 더티리드를 막습니다. 다만 같은 트랜잭션 안에서 여러 번 조회하면, 그 사이 다른 트랜잭션이 커밋한 최신값이 매번 다시 보일 수 있어 반복불가읽기/팬텀리드는 여전히 발생 가능합니다. 대부분의 일반적인 CRUD 작업(단순 조회, 단건 등록/수정)은 이 수준으로 충분합니다.

> 프로젝트 예시: 자전거 상세 조회, 결제내역 조회처럼 "한 번 읽고 끝나는" 단순 쿼리는 READ COMMITTED로 아무 문제 없음.

**3. REPEATABLE READ (반복 읽기 시 값 고정)**

트랜잭션이 시작된 시점의 스냅샷을 트랜잭션이 끝날 때까지 계속 보게 해서, 같은 트랜잭션 안에서 몇 번을 다시 읽어도 항상 같은 값을 봅니다.

> 프로젝트 예시: 대여 시작~반납까지 하나의 트랜잭션으로 묶어 요금정책을 계산한다면, 그 사이 관리자가 요금정책을 바꿔도 이 트랜잭션은 시작 시점의 요금정책만 일관되게 보게 하고 싶을 때 사용.

**4. SERIALIZABLE (완전 순차 실행과 동일한 결과 보장) — 가장 엄격**

동시에 실행되는 트랜잭션들이 마치 한 줄로 순서대로 실행된 것과 동일한 결과가 나오도록 보장하는 수준. PostgreSQL은 SSI(Serializable Snapshot Isolation) 방식으로 구현되어 있어, 락으로 무작정 막는 대신 동시 실행은 허용하되 "이 순서로 실행됐다면 문제가 생겼을 조합"을 감지해서 그 트랜잭션만 실패시킵니다(`could not serialize access` 에러 발생 → 애플리케이션이 재시도해야 함).

> 프로젝트 예시: 정산 배치처럼 "절대 숫자가 어긋나면 안 되는" 마감 처리에 선택적으로 사용할 수 있으나, 일반적인 서비스 트랜잭션(대여, 결제)에는 재시도 로직 부담 때문에 잘 안 씀. 자전거 이중 대여 방지 같은 문제는 격리수준을 올리는 것보다 `SELECT ... FOR UPDATE` 같은 명시적 락으로 해결하는 게 실무적으로 더 흔함.

### 세 가지 이상현상(Anomaly) 자세히

격리수준이 낮을 때 생길 수 있는 문제들이고, 격리수준이 올라갈수록 아래 순서대로 하나씩 막아줍니다.

**1. 더티 리드 (Dirty Read) — "아직 확정 안 된 걸 읽어버림"**

한 트랜잭션이 **아직 커밋(확정)하지 않은** 데이터를 다른 트랜잭션이 읽어버리는 현상. 원본 트랜잭션이 나중에 **롤백**되면, 방금 읽은 값은 애초에 존재한 적 없던 값이 되는데, 이미 그 값을 읽어서 뭔가 처리해버린 뒤라는 게 문제입니다.

> 프로젝트 예시: 결제 트랜잭션이 `TB_PAYMENT`에 "결제완료"를 잠깐 써놓고 자전거 잠금해제를 호출하려는 순간 카드사 승인이 실패해 롤백한다고 하면, 그 짧은 순간 다른 프로세스가 "결제완료" 상태를 읽고 자전거를 이미 대여완료 처리해버릴 수 있음. 결제는 취소됐는데 자전거는 나가버리는 상황.

**2. 반복 불가능한 읽기 (Non-repeatable Read) — "같은 걸 두 번 읽었는데 값이 바뀜"**

**하나의 트랜잭션 안에서** 같은 행을 두 번 읽었는데, 그 사이에 **다른 트랜잭션이 그 값을 커밋**해버려서 값이 달라지는 현상. 더티리드와 다른 점은, 여긴 이미 "커밋된" 값을 읽는 거라 그 자체는 정상인데, "같은 트랜잭션 안에서 값이 왔다갔다 한다"는 게 문제입니다.

> 프로젝트 예시: 대여 시작 시점에 `TB_FEE_POLICY`에서 요금을 한 번 읽고, 반납 시점에 최종 정산을 위해 다시 읽었는데, 그 사이 관리자가 요금정책을 수정·커밋해버리면 같은 대여 건인데 시작할 때 본 요금과 끝날 때 본 요금이 달라짐.

**3. 팬텀 리드 (Phantom Read) — "같은 조건으로 조회했는데 행 개수(집합)가 달라짐"**

반복 불가능한 읽기와 비슷하지만 차이는, "이미 있던 행의 값이 바뀐 것"이 아니라 **"새로운 행이 나타나거나 사라진 것"**이라는 점. 마치 유령처럼 없던 행이 나타난다고 해서 팬텀이라고 부릅니다.

> 프로젝트 예시: "이 서비스권역 안에서 대여 가능한 자전거 개수"를 트랜잭션 안에서 두 번 세었는데, 그 사이 다른 사용자가 그 지역 자전거 하나를 대여해 커밋해버리면 첫 번째 카운트와 두 번째 카운트의 "결과 집합 크기" 자체가 달라짐.

**세 현상 비교**

| 현상 | 무엇이 문제인가 | 원인이 되는 다른 트랜잭션의 동작 |
|---|---|---|
| 더티 리드 | 커밋 안 된(불확정) 값을 읽음 | 아직 커밋도 안 했는데 값이 보임 |
| 반복 불가능한 읽기 | 같은 행의 값이 두 번 읽을 때마다 달라짐 | UPDATE 후 커밋 |
| 팬텀 리드 | 같은 조건인데 행의 개수(집합)가 달라짐 | INSERT/DELETE 후 커밋 |

> **참고**: PostgreSQL의 REPEATABLE READ는 표준 규격상으로는 팬텀리드까지 막을 의무가 없지만, MVCC 스냅샷 방식(트랜잭션 시작 시점의 스냅샷을 끝까지 유지)으로 구현되어 있어 실질적으로 팬텀리드까지 막아주는 보너스 효과가 있습니다.

### 낙관적 락 vs 비관적 락

| | 낙관적 락(Optimistic Lock) | 비관적 락(Pessimistic Lock) |
|---|---|---|
| 방식 | 버전 컬럼(예: `UPD_DTTM`, `VERSION`)을 두고, UPDATE 시 "내가 읽은 버전과 같을 때만" 갱신 | `SELECT ... FOR UPDATE`로 미리 행을 잠금 |
| 충돌 처리 | 충돌 시 갱신 실패 → 애플리케이션이 재시도 | 다른 트랜잭션은 락이 풀릴 때까지 대기 |
| 적합한 상황 | 충돌이 드문 경우 (동시 수정 가능성 낮음) | 충돌이 잦은 경우 (동시 접근 확실) |
| 프로젝트 적용 후보 | 배터리 상태, 자전거 정보 수정 등 | **자전거 대여 시작 시점** (동시 대여 시도 방지에 적합) |

> **질문 대비**: "낙관적 락은 왜 대여에 안 쓰나?" → 낙관적 락은 충돌 시 "실패 후 재시도"인데, 자전거 대여처럼 "먼저 온 사람이 이겨야 하고 진 사람에게 즉시 실패를 알려야 하는" 상황엔 비관적 락(즉시 대기/차단)이 사용자 경험상 더 적합.

### 제3의 패턴 — 원자적 조건부 갱신 (Atomic Conditional Update)

실무에서 자전거 대여처럼 "상태값 하나를 두고 먼저 온 요청만 성공시키고 싶은" 경우, 낙관적/비관적 락 어느 쪽도 아닌 훨씬 간단한 방법을 씁니다.

```sql
UPDATE tb_bicycle
SET bicycle_stat_cd = 'RENTED'
WHERE bicycle_id = 123 AND bicycle_stat_cd = 'AVAILABLE';
```

- SELECT로 먼저 읽고 판단한 뒤 UPDATE를 따로 날리는 게 아니라, **조건 검사와 갱신을 한 문장(원자적 연산)으로 합침**
- 두 트랜잭션이 동시에 이 문장을 날리면, 먼저 도달한 것만 실제로 상태를 바꾸고 커밋되며, 늦게 도달한 트랜잭션은 그 값을 다시 평가했을 때 조건(`= 'AVAILABLE'`)이 이미 깨져 있어 **영향받은 행 수(affected rows) 0건으로 실패** — 애플리케이션은 이 0건을 보고 실패 처리
- 별도의 버전 컬럼(낙관적 락)도, 미리 걸어두는 긴 락(비관적 락)도 필요 없음. 업무상 이미 존재하는 상태값이 버전 역할을 자연스럽게 겸함

> **질문 대비**: "이것도 락을 거는 거 아닌가?" → 모든 UPDATE 문은 실행되는 그 찰나 항상 행 락을 겁니다(DB의 기본 동작). 다만 "비관적 락"이라 부르는 건 **읽는 시점부터 의도적으로 락을 오래 붙잡아두는 전략**을 가리키는 것이지, "락이 걸린다"는 사실 자체를 가리키는 게 아닙니다. 이 방식은 읽기-판단-쓰기 사이의 간격이 없어 그런 전략적 선택 자체가 필요 없는, 제3의 접근입니다.

## 9. 인덱스 심화

### 인덱스 종류와 용도

| 종류 | 원리 | 적합한 경우 | 프로젝트 적용 |
|---|---|---|---|
| B-Tree | 정렬된 트리 구조, 등호/범위/정렬 모두 지원 | 범용, 대부분의 경우 | 고객ID, 대여ID 등 FK |
| Unique B-Tree | B-Tree + 유일성 제약 | PK, 대체키 | CUST_PUBLIC_ID |
| GiST | 일반화된 트리, 공간/범위 데이터 지원 | 공간 검색(Point/Polygon) | 자전거위치, 서비스권역 |
| GIN | 역색인(inverted index), 다중값 검색 | 배열, 전문검색(Full-text) | (본 프로젝트엔 미사용, 참고용) |
| BRIN | 블록 단위 근사 최소/최대값만 저장, 매우 작음 | 삽입 순서와 값이 상관관계 있는 대용량 컬럼 | 대여시작일시, 결제승인일시 |
| Hash | 정확 일치(=)만 지원, 범위 검색 불가 | 매우 드물게 사용 | 미사용 |

### 부분 인덱스(Partial Index) vs 복합 인덱스(Composite Index)

- **부분 인덱스**: `WHERE 조건`을 인덱스 정의에 포함 → 희소 데이터에 효과적 (예: `시설ID IS NOT NULL`)
- **복합 인덱스**: 여러 컬럼을 묶어서 하나의 인덱스로 (예: `(고객ID, 결제요청일시 DESC)`) → **컬럼 순서가 중요**: 앞쪽 컬럼이 검색조건(=), 뒤쪽이 정렬/범위에 쓰이는 게 일반적

> **질문 대비**: "복합 인덱스 컬럼 순서를 바꾸면 어떻게 되나?" → `(결제요청일시, 고객ID)` 순서면 "특정 고객의 결제내역" 조회에는 안 쓰이거나 비효율적임. 카디널리티 높은(분포가 다양한) 컬럼이나 실제 WHERE 조건의 등호 비교 컬럼을 앞에 두는 게 일반적 원칙.

### 인덱스를 걸지 않는 경우 (분포도/선택도)

- 선택도(Selectivity) = 조건을 만족하는 행 수 / 전체 행 수. **선택도가 낮을수록(=매칭 행이 적을수록) 인덱스 효과가 큼**
- 분포도가 높은 컬럼(값 종류는 적은데 각 값이 많은 행에 걸침, 예: 요금정책ID)은 인덱스가 있어도 옵티마이저가 풀스캔을 택하고, 쓰기 비용만 낭비 → 인덱스 안 거는 게 유리

## 10. 파티셔닝 vs 샤딩 vs 수직분할 — 용어 구분 (앞서 논의했지만 정리)

| 용어 | 정의 | 프로젝트 예시 |
|---|---|---|
| **수평 분할(파티셔닝, Partitioning)** | 같은 스키마의 테이블을 행(row) 기준으로 여러 물리 테이블에 분산 (한 DB 서버 내) | (미적용) 결제/로그 테이블을 월별로 나누면 이에 해당 |
| **수직 분할(Vertical Partitioning)** | 한 테이블의 컬럼을 여러 테이블로 분리 | 자전거 ↔ 자전거위치 분리 |
| **샤딩(Sharding)** | 파티셔닝을 **여러 물리 서버(노드)**로 분산 | (미적용) 서비스권역이 여러 도시로 확장되면 지역별 샤딩 고려 가능 |
| **레플리케이션(Replication)** | 같은 데이터를 여러 서버에 복제 (분할이 아니라 복사) | 읽기 부하 분산용 read replica |

> **질문 대비**: "서비스가 전국 확장되면 어떻게 스케일링할 건가?" → "서비스권역(지역) 기준으로 샤딩하거나, 읽기 트래픽이 많은 자전거 위치조회는 read replica로 분산하는 걸 고려할 수 있다"고 답하면 확장성 고민을 보여줄 수 있음.

## 11. 로그/이력 데이터 저장 전략 (발표에 있던 내용 정리)

| 저장소 | 특성 | 적합한 데이터 | 프로젝트 적용 |
|---|---|---|---|
| RDB (일반 테이블) | 트랜잭션/무결성 보장, 복원 가능 | 복원이 필요한 핵심 이력 | 대여, 결제 내역 |
| TimescaleDB | 시계열 데이터 특화 (시간 기준 파티셔닝 내장, 압축) | 시간에 따른 연속적 변화 분석 | 자전거 위치 변경 이력 |
| ElasticSearch | 역색인 기반, 다양한 조건의 전문검색/필터링에 강함 | 다양한 축으로 검색되는 일반 로그 | 자전거/배터리 상태변경 로그, 서비스권역 로그 |

> **질문 대비**: "왜 위치이력은 RDB가 아니라 TimescaleDB인가?" → 위치이력은 초 단위로 계속 쌓이는 대용량 시계열 데이터라 일반 RDB 테이블에 넣으면 인덱스/파티션 관리 부담이 크고, TimescaleDB는 시간 기준 자동 파티셔닝(청크)과 압축을 기본 제공해서 대용량 시계열에 최적화되어 있음.

## 12. 데이터 무결성 제약(Constraint) 종류

| 제약 | 역할 | 프로젝트 예시 |
|---|---|---|
| PRIMARY KEY | 유일성 + NOT NULL | `RENT_ID` |
| FOREIGN KEY | 참조 무결성 (부모에 존재하는 값만 허용) | `TB_RENTAL.CUST_ID` |
| UNIQUE | 유일성만 (NULL 허용 가능) | `CUST_EMAIL` |
| NOT NULL | 필수값 강제 | `CUST_NM` |
| CHECK | 값의 범위/조건 강제 | 예: `할인율 BETWEEN 0 AND 100` |
| DEFAULT | 값 미지정 시 기본값 | `REG_DTTM DEFAULT now()` |

> **질문 대비**: "쿠폰의 배타적 서브타입을 DB가 강제하나?" → 기본적으로 안 됨. `CHECK` 제약이나 트리거로 별도 구현해야 하며, 지금 구조는 애플리케이션 레벨 검증에 의존하고 있다는 점을 솔직히 인정하고 "개선한다면 트리거나 배타적 CHECK 제약 추가를 고려할 수 있다"고 답하면 됨.

## 13. NoSQL vs RDB 선택 기준 (일반 지식, 나올 수 있는 질문)

| 기준 | RDB가 유리 | NoSQL이 유리 |
|---|---|---|
| 데이터 관계 | 여러 엔터티 간 관계가 복잡, JOIN 필요 | 관계 없이 독립적인 문서/키-값 |
| 트랜잭션 | ACID 보장 필요 (결제, 재고 등) | 최종 일관성(eventual consistency)으로 충분 |
| 스키마 | 구조가 명확하고 자주 안 바뀜 | 스키마가 유동적/가변적 |
| 확장 | 수직 확장 위주 (샤딩은 복잡) | 수평 확장이 쉬움 |

> **질문 대비**: "왜 전체를 RDB(PostgreSQL)로만 설계했나?" → "결제/대여처럼 금전과 직결되고 관계가 복잡한 트랜잭션 데이터는 ACID 보장이 필수라 RDB가 적합하고, 로그성 데이터는 이미 목적별로 ElasticSearch/TimescaleDB로 분리해서 각 데이터 특성에 맞는 저장소를 선택했다"고 답하면 일관된 논리가 됩니다.

## 14. 쿼리 실행계획 (EXPLAIN) — 알아두면 좋은 기본 개념

- `EXPLAIN`: 실제 실행 안 하고 계획만 보여줌
- `EXPLAIN ANALYZE`: 실제로 실행해서 진짜 소요시간까지 보여줌
- 주요 스캔 방식
  - **Seq Scan**: 테이블 전체 순차 스캔
  - **Index Scan**: 인덱스로 찾고 heap에서 실제 값 가져옴
  - **Index Only Scan**: 인덱스만으로 응답 완성 (heap 접근 없음, 커버링 인덱스 효과)
  - **Bitmap Heap Scan**: 여러 인덱스 결과를 비트맵으로 합쳐서 처리 (조건이 여러 개일 때)

> **질문 대비**: "인덱스를 걸었는데 안 쓰이면 어떻게 확인하나?" → `EXPLAIN ANALYZE`로 실행계획을 확인하고, 통계정보(`ANALYZE 테이블명`)가 최신인지 먼저 점검한다고 답하면 실무 감각을 보여줄 수 있음.

## 15. 이 프로젝트에서 특히 날카롭게 나올 수 있는 통합 질문 몇 가지

1. **"고객 탈퇴 후 같은 이메일로 재가입하면 UNIQUE 제약과 충돌 안 하나?"**
   → Soft delete(`WITHDRAWN_DTTM`) + UNIQUE 제약의 전형적 충돌 지점. 실무적으로는 부분 유니크 인덱스(`UNIQUE ... WHERE WITHDRAWN_DTTM IS NULL`)로 "활성 회원 중에서만 유일" 하도록 처리하는 방법이 있음.

2. **"자전거 상태값(대여가능/대여중/정비중 등)을 동시에 여러 프로세스가 바꾸려 하면?"**
   → 위 8번(동시성 제어) 답변과 연결. `SELECT ... FOR UPDATE` 혹은 원자적 조건부 UPDATE(`WHERE 상태='AVAILABLE'`)로 상태 전이(state transition) 시점에 방어.

3. **"인센티브 정책이 실제로 어느 테이블과 연결되는가?"**
   → ERD상 FK 연결이 명확하지 않은 부분이라 **발표 전 반드시 직접 확인/정리해두세요.**

4. **"만약 서비스 권역이 여러 도시로 늘어나면 스키마를 얼마나 고쳐야 하나?"**
   → 이미 `TB_SERVICE_REGION`이 마스터로 분리되어 있어 신규 지역 추가는 데이터 INSERT만으로 가능 (스키마 변경 불필요)하다는 점을 강점으로 설명 가능.
